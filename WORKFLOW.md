# Signal Deck — Workflow Ritual

## Purpose

This file exists to keep Telegram, Mac-side work, future sessions, and any sub-agents aligned around one shared project reality.

## Canonical Project Home

`~/Desktop/Signal Deck`

All active Signal Deck work should happen here or point here.

## Canonical Identity

**Project name:** Signal Deck

Do not rename it casually in conversation or across files.
Avoid parallel prototype names that split the project identity.

## Required Start-of-Work Ritual

Before doing meaningful work on Signal Deck:

1. Read `PROJECT.md`
2. Read `STATUS.md`
3. Read `NEXT.md`
4. Confirm work is happening inside `~/Desktop/Signal Deck`

## Required End-of-Work Ritual

After meaningful build progress:

1. Update `STATUS.md` if the actual state changed
2. Update `NEXT.md` if priorities or next actions changed
3. Keep new files inside the canonical project folder whenever possible

## Build Mode Rule

Signal Deck is an active software build project.
Do not treat it as only a concept, moodboard, or brainstorming object.
Bias toward execution.

## Simplicity Rule

Do not let the project bloat into fake artist SaaS.
Keep it:
- song-first
- local-first
- simple
- functional
- generic/reusable for musicians

## Data Rule

Do not assume Zach-specific demo content should remain the default app state.
The product should move toward:
- blank start
- local save/load
- workspace-based use
- import/export later

## Visual Rule

Use Zach’s Boxman and Martian drawings as a brand/world layer.
Do not let them overpower usability.
Clean product UI first. Weird little universe second.

## Drift Prevention Rule

If there is ever confusion between chats, folders, or builds:
- `PROJECT.md` defines identity
- `STATUS.md` defines current reality
- `NEXT.md` defines immediate priorities
- this file defines the ritual

These files are the source of truth.
